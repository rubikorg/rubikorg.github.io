Project Structure

Main File

After generating your Rubik workspace, you can view cmd/server/main.go file which is the heart of Rubik server.

package main

import (
    cfg "myproject/cmd/server/config"
    "myproject/cmd/server/routers"
    "github.com/rubikorg/rubik"
)

func main() {
    var config cfg.ProjectConfig
    err := rubik.Load(&config)
    if err != nil {
        panic(err)
    }

    routers.Import()
    panic(rubik.Run())
}

Let's break down the lines of this main() func. The rubik.Load() func initializes configuration of your server and feeds it to Rubik runtime, you'll learn more about it in this Configuration chapter.

routers.Import() imports all your rubik.Router which is inside routers directory. rubik.Run() runs your server, if we dive a little deeper it somewhat works like an OS booting all your routes and their Handler Funcs.

Config Folder

Config folder holds your Rubik config. Unlike any other Web frameworks Rubik implements it's own configuration consumer, making it easier to focus only on your route implementations. It is a seamless integration and uses TOML as it's default configuration language. More information can be found in the Configuration chapter.

Routers Folder

Routers folder holds a group of routes that can be tied together in the form of rubik.Router. Routers can hold multiple routes and rubik holds all routers which are added inside routers/import.go. The Import() that we saw in the above section is responsible for serving all your routes to Rubik. You can think of each folder inside routers as a separate domains to decouple your logic separately.

Example: players, users, teams can each be separate routers.

:: route.go

package teams

import (
    r "github.com/rubikorg/rubik"
)

// Router is index's router
var Router = r.Create("/teams")

var playersRoute = r.Route{
    Path:  "/players",
    Controller: playersCtl,
}

func init() {
    Router.Add(playersRoute)
}
  • A new Router is initialized by using r.Create() func.
  • r.Route{} defines a new rubik Route. This route takes in Path which is appended to Router's path, in our case you will be able to access this route at /teams/players. Controller holds your main business logic and is abstracted inside controller.go file.
  • Our playersRoute is then added to the rubik Router by calling Router.Add() func.

:: controller.go

package teams

import (
    r "github.com/rubikorg/rubik"
)

func playersRoute(en interface{}) r.ByteResponse {
    return r.Success("Yo", r.Type.Text)
}
  • There is nothing going on here except the r.Success() func. It accepts an interface, and encoding type. There is also r.Failure() method to denote API failures.

Note

You dont need to write any of this router code as okrubik generates them for you. Take a look at Tooling::gen::router section.

Static Folder

If static folder is present inside your root of the binary folder Rubik boots another route /static which serves the whole static folder

Templates Folder

Anything inside the templates folder is accessible to the rubik.Render() func. Simply calling the below code:

import r "github.com/rubikorg/rubik"

func myCtl(en interface{}) r.ByteResponse {
    return r.Render(r.Type.HTML, nil, "index.html")
}

will render templates/index.html as HTML response for your route.

Storage Folder

The storage folders can be viewed as containers for storing your files. The Storage APIs provides easy to use utility functions for storing, removing and fetching files from Storages.

import r "github.com/rubikorg/rubik"

func myCtl(en interface{}) r.ByteResponse {
    paymentStore := r.Storage.Access("payment")
    contentBytes := paymentStore.Get("myfile.txt")
    if contentBytes != nil {
        return r.Success(contentBytes, r.Type.Text)
    }

    return r.Failure(404, r.E("File not found"))
}
